# ANA 구축 절차 (전문)

Agent Native App을 처음 짓거나 기존 ANA를 진화시키는 단계별 절차. SKILL.md의 요약을 확장한 실행 가이드다. 상위 하네스 일반론은 github.com/revfactory/harness의 6단계 워크플로우를 참조한다.

## 목차
- Phase 0: 컨텍스트 확인
- Phase 1: 도메인 1화면 정의 (Watch)
- Phase 2: 얼굴 입히기 (uxui-design-system)
- Phase 3: 신경계 배선 (fakechat-dashboard-agent)
- Phase 4: 말로 키운다 (진화 루프)
- Phase 5: 소유·공개
- 테스트 시나리오

## Phase 0: 컨텍스트 확인 (초기/후속/부분 재실행 판별)

워크플로우 시작 시 대상 디렉터리의 기존 산출물을 확인해 실행 모드를 결정한다.

- 대시보드 서버(`server.js`), 상태 JSON, 대시보드 HTML이 **없음** → **초기 구축** (Phase 1~5 전체).
- **있음 + 사용자가 부분 수정 요청**("음성 입력 추가", "30분 단위로", "버튼 추가") → **부분 재실행**: Phase 4 진화 루프로 직행, 해당 부분만 변경.
- **있음 + 새 도메인** → **새 ANA**: 기존을 보존하고 별도 인스턴스로 구축.

기존 산출물이 있으면 먼저 읽고, 사용자 피드백이 주어지면 그 부분만 수정한다(전체 재생성 금지).

## Phase 1: 도메인 1화면 정의 (Watch)

매일 봐야 할 **핵심 상태 1개**를 정한다.

1. "매일 보는 건 정해져 있다" — 일정/지표/주문/작업 큐 중 무엇인가? 하나로 좁힌다.
2. 그 1화면에 들어갈 **핵심 엔티티**를 결정한다(예: 주간 계획표 = 시간 × 요일 × 가족).
3. 데이터 모델 = 핵심 엔티티 + **`version` 필드**(동기화용). JSON 파일로 시작.
4. 산출 기준: "이 화면 하나를 매일 본다"가 한 문장으로 말해지면 통과.

> 좁히지 못하면 ANA가 흐려진다. 여러 화면이 떠오르면 가장 자주 보는 1개부터.

## Phase 2: 얼굴 입히기 (`uxui-design-system`)

`skills/uxui-design-system/SKILL.md`를 읽고 그 절차로 대시보드의 시각 맥락을 만든다.

- **토큰 우선**: 색/타이포/라운드/그림자/간격을 `:root` CSS 변수로. 하드코딩 금지(다크모드 자동 대응). `references/design-tokens.css`를 출발점으로.
- **큰 숫자·굵은 글씨·좁은 자간**: 핵심 지표는 크고 굵게(800), 자간 -0.03~-0.04em. 브랜드 블루로 강조.
- **부드러운 면**: 흰 카드 + 옅은 배경 + 소프트 섀도우 2겹 + 16~24px 라운드.
- **하단 액션 바**: `[액션][대화하기]` 버튼을 신설(예: `[스케줄등록][대화하기]`). 바텀시트/팝업 패턴은 `references/bottom-bar-and-sheets.md`.
- **모바일 우선**: `dvh`, `env(safe-area-inset-*)`, `VisualViewport`(키보드), 터치 타깃 ≥44px.

산출 기준: 핵심 상태가 1화면에 한눈에 보이고, 모든 색/라운드가 토큰을 참조한다.

## Phase 3: 신경계 배선 (`fakechat-dashboard-agent`)

`skills/fakechat-dashboard-agent/SKILL.md`를 읽고 그 절차로 watch+converse 배선을 만든다.

1. **데이터 모델 확정**: Phase 1의 엔티티 + `version`. JSON 파일.
2. **서버**: `references/server.js`를 복사해 스키마/엔드포인트를 맞춘다. 핵심 API — 정적 서빙, `/api/chat`(인박스), `/api/feed`(데이터+version), `/api/agent`(리치 응답), 제안/승인.
3. **인바운드 릴레이**: `references/fakechat-bridge.js`로 새 채팅 요청을 fakechat WS에 밀어 Claude 세션에 자동 도착시킨다.
4. **fakechat 채널**: 메시지를 Claude 세션에 `<channel source="fakechat">`로 푸시.
5. **응답 규칙**: 변경은 채팅 텍스트가 아니라 **전/후 미리보기 + 승인 카드**(`/api/agent`)로. 인바운드만 릴레이, 응답은 항상 `/api/agent`.
6. **동기화**: `version` 폴링 2.5초 병행. 완료/토글은 서버 저장. SSE만 믿지 말 것.

산출 기준: 사용자가 화면을 보며 채팅하면 메시지가 Claude 세션에 자동 도착하고, 응답이 승인 카드로 표시된다.

## Phase 4: 말로 키운다 (Agent as Runtime — 진화 루프)

앱을 쓰는 도중 **요청만으로** 기능을 키운다. 이게 ANA의 진짜 힘이다.

- 사용자의 자연어 요청 → 에이전트가 변경안을 **전/후 미리보기 + 승인 카드**로 제시 → 승인 시 코드/데이터 반영 → 모든 기기 동기화.
- 실제 진화 예:
  - "음성으로 입력되게" → 🎙️ 마이크(Web Speech) 추가
  - "30분 단위로, 한 줄·고정 높이" → 격자 규칙 변경
  - "이전 5일·앞으로 5일" → 롤링 윈도우
  - "하단에 [스케줄등록][대화하기]" → 액션 바 신설
  - "9시 반까지 늘려줘" → 표시 범위 확장
- 개발 사이클이 아니라 **대화 한 번**. 변경은 항상 미리보기→승인으로 안전하게.

같은 유형의 요청이 2회 이상 반복되면 빌딩블록 스킬 자체에 반영을 제안한다(진화).

## Phase 5: 소유·공개 (Own Your Harness)

- **셀프호스팅 유지**: 의존성 0, 종속 없음, 이식 가능.
- **외부 공개**: 고정 주소 터널(cloudflared 등)로 공개. 터널이 `text/event-stream`을 버퍼링하므로 SSE만 믿지 말고 폴링 병행.
- **무인증/개인 운영**: 휴대폰만 있으면 어디서든 운영.

## 테스트 시나리오

**정상 흐름:** 도메인 "우리집 주간 계획표"
1. Phase 1 — 시간 × 요일 × 가족 4명 격자 + version.
2. Phase 2 — 토스 스타일 격자 대시보드 + 하단 `[스케줄등록][대화하기]`.
3. Phase 3 — 서버·릴레이·채널 연결. "민준 7시 영어 추가" 채팅 → Claude 세션 도착.
4. 응답 — `schedule.json` 읽고 전/후 미리보기 + 승인 카드 → 승인 시 반영, 모든 기기 동기화.
5. Phase 4 — "음성으로 입력되게" → 마이크 추가(대화 한 번으로).

**에러 흐름:** 터널 뒤에서 SSE가 버퍼링되어 응답이 안 보임
- 폴링(2.5초) 병행이 누락된 경우다. 수신부에 version 폴링을 추가하고 메시지 id로 중복 무시 → 모든 접속자 동기화 복구.

**검수:** SKILL.md의 "검수 체크리스트"로 Watch/Converse/Runtime/동기화/Own 7항목을 확인한다.
