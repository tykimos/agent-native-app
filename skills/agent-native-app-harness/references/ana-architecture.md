# ANA 아키텍처 — 구성도와 설계 원리

Agent Native App의 런타임 구조, 데이터 흐름, 그리고 "보면서 대화"가 어떻게 배선되는지를 상세히 설명한다. 빌딩블록 스킬(`uxui-design-system`, `fakechat-dashboard-agent`)이 각 구성요소를 어떻게 채우는지 매핑한다.

## 목차
1. 전체 구성도
2. 구성요소별 책임
3. 데이터 흐름 (인바운드/응답)
4. 다기기 동기화
5. "내가 앱에 맞추는" 구조 vs ANA
6. 빌딩블록 ↔ 아키텍처 매핑

## 1. 전체 구성도

```
   사용자(휴대폰/PC)
        │  ① 대시보드를 보며   ② 대화로 요청
        ▼
 ┌──────────────────────┐      ┌───────────────────────────┐
 │  대시보드 (웹)        │◀────▶│  채널 (fakechat 등)        │
 │  · 핵심 상태 한눈에   │ 푸시 │  · 메시지를 에이전트로 전달 │
 │  · 채팅 입력/응답     │      └─────────────┬─────────────┘
 └──────────┬───────────┘                    ▼
            │  데이터 API           ┌───────────────────────────┐
            └──────────────────────▶│  코딩 에이전트(Claude Code)│
              읽기/제안/적용         │  · 상태 읽기 · 제안·적용     │
                                    │  · 앱 코드까지 수정         │
                                    └───────────────────────────┘
```

핵심 규칙: **인바운드는 채널로 자동 푸시, 응답은 대시보드로(제안→승인)**. 그래야 watch(시각 맥락) + converse(대화)가 한 화면에서 유지된다.

## 2. 구성요소별 책임

| 구성요소 | 책임 | 빌딩블록 |
|---|---|---|
| **대시보드(웹)** | 핵심 상태를 1화면에 시각화 + 채팅 입력/리치 응답 표시 | `uxui-design-system`(얼굴) |
| **채널(fakechat)** | 사용자의 인바운드 메시지를 Claude 세션에 자동 푸시("확실한 배달부") | `fakechat-dashboard-agent`(신경계) |
| **대시보드 서버** | 정적 대시보드 서빙 + 채팅 브리지 API(인박스/피드/제안/승인) + 상태 JSON, 의존성 0 | `fakechat-dashboard-agent` |
| **인바운드 릴레이** | 대시보드의 새 채팅 요청을 fakechat WS로 밀어 Claude 세션에 자동 도착 | `fakechat-dashboard-agent` |
| **코딩 에이전트(두뇌)** | 채널 메시지 수신 → 데이터 API 읽기 → 리치 응답(`/api/agent`) → **앱 코드 수정** | Claude Code 런타임 |

## 3. 데이터 흐름

### 인바운드 (사용자 → 에이전트)
```
[웹 대시보드] ──사용자 채팅──► /api/chat ──► inbox(요청 큐)
                                              │
                              인바운드 릴레이가 /api/inbox-wait 롱폴
                                              │ → ws.send → fakechat 채널
                                              ▼
                              fakechat WS ──notification──► Claude 세션(두뇌)
```

### 응답 (에이전트 → 사용자)
```
Claude 세션 ──POST /api/agent(리치 응답: 텍스트/제안 diff)──► 대시보드에 표시
```

설계 포인트: **인바운드만 릴레이로 보내고 응답은 항상 `/api/agent`로** 보낸다. 그래야 제안/승인 카드 같은 리치 UX가 유지되고, 채널은 "푸시 배달"만 담당한다. fakechat의 user 주입은 채널 UI로 echo되지 않아 중복이 없다.

## 4. 다기기 동기화

- 상태 파일에 **`version`** 필드를 두고 변경마다 +1.
- 대시보드는 피드 응답에 `version`을 함께 받아 **2.5초 폴링**으로 버전 변화 감지 시 데이터 재요청 → 모든 접속자 자동 동기화.
- 완료/토글 등 사용자 액션은 **localStorage가 아니라 서버에 저장**해야 전원에게 반영된다.
- SSE는 저지연이지만 **cloudflared 등 터널이 `text/event-stream`을 버퍼링**하므로, 폴링을 항상 병행한다(SSE만 믿지 말 것). 수신부는 메시지 id로 중복 무시.

## 5. "내가 앱에 맞추는" 구조 vs ANA

| 축 | 전통 SaaS | 챗봇 | 노코드 | **ANA** |
|---|---|---|---|---|
| 시각 맥락 | O(고정) | X | O(템플릿) | **O(내 대시보드)** |
| 자연어 운영 | X | O | △ | **O** |
| 앱을 그 자리서 변경 | X | X | △ | **O(에이전트가 코딩)** |
| 내 방식에 수렴 | X | △ | △ | **O** |
| 소유·이식 | X | X | X | **O(셀프호스팅)** |

전통 앱/챗봇/노코드는 모두 "내가 앱에 맞추는" 구조. ANA는 **앱이 나에게 맞춰 자란다**.

## 6. 빌딩블록 ↔ 아키텍처 매핑

- **얼굴(Watch)** = `uxui-design-system`: 대시보드의 토큰·레이아웃·하단 액션 바·바텀시트. 핵심 지표를 한눈에.
- **신경계(Converse)** = `fakechat-dashboard-agent`: 서버·릴레이·채널·세션 배선. 인바운드 자동 푸시 + 리치 응답.
- **두뇌(Runtime)** = Claude Code 세션: 읽기→제안→적용, 그리고 앱 코드 수정.

새 구성요소(인증, 알림, 배포 터널 등)가 반복되면 같은 패턴의 빌딩블록 스킬로 `skills/`에 추가하고 이 매핑 표를 갱신한다.
